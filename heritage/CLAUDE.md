# CLAUDE.md (heritage)

이 파일은 **여러 프로젝트에서 재사용 가능한 개발 원칙**을 모아 둔 기본 CLAUDE.md 다.
새 프로젝트에 들어갈 때 이 파일을 그대로 복사하거나, 프로젝트 고유 섹션(스택·도메인 모델·디렉토리 구조)만 위에 추가해 사용한다.

원칙은 도구·언어 중립적으로 기술되며, 명령어가 필요한 곳에는 **프로젝트 어댑터 변수**(섹션 마지막)를 사용한다.

---

## 개발 철학: TDD 필수

이 원칙을 채택한 모든 프로젝트의 **프로덕션 코드는 TDD 기반**으로 작성한다.
TDD는 단순히 "테스트 코드를 먼저 쓰는 것"이 아니라, **RED 단계에서 시스템이 마주칠 모든 edge case 를 선제적으로 설계하고 커버하는 디자인 활동**이다.

### 사이클: RED → GREEN → REFACTOR

#### 1. RED — "실패하는 테스트를 먼저 쓴다"
- **목적**: 테스트 작성 자체가 아니라 **요구사항·시스템 경계·edge case 의 발굴**.
- **해야 할 것**:
  - Happy path뿐 아니라 **edge case 를 적극적으로 발굴**한다 (아래 체크리스트).
  - 각 edge case 마다 **명시적으로 실패하는 테스트**를 작성한다.
  - 테스트는 반드시 **먼저 실패(RED)** 해야 한다. 통과한다면 테스트가 틀렸거나 구현이 이미 있다는 뜻.
- **하지 말 것**:
  - "나중에 edge case 추가하자"며 happy path만 쓰고 다음 단계로 넘어가지 않는다.
  - 프로덕션 코드를 먼저 생각하고 거기에 맞춰 테스트를 쓰지 않는다 (역방향 TDD 금지).

> **TDD Rule**: 실패하는 테스트를 보지 않은 채 구현을 시작하지 않는다. 실패를 보지 못했다면 TDD 가 아니다.

#### 2. GREEN — "테스트를 통과시키는 최소 코드를 쓴다"
- **목적**: 오직 RED 테스트들을 통과시키는 것만.
- **해야 할 것**:
  - 가능한 한 단순하게, 직관적인 구현.
  - 실패 테스트가 없는 기능은 이 단계에서 추가하지 않는다.
- **하지 말 것**:
  - 미래를 위한 추상화, 사용되지 않는 매개변수, 아직 테스트 없는 분기 추가 금지.

#### 3. REFACTOR — "동작을 바꾸지 않고 구조를 개선한다"
- **목적**: 테스트가 초록불인 상태에서 **내부 구조만** 다듬는다.
- **해야 할 것**:
  - 중복 제거, 명명 개선, 함수/클래스 분리, 매직 넘버 상수화.
  - 리팩터 중 **모든 테스트는 계속 통과**해야 한다. 한 번이라도 깨지면 즉시 되돌린다.
- **하지 말 것**:
  - 기능 추가. 새 동작이 필요하면 사이클을 처음(RED)부터 다시 돈다.
  - 테스트가 없는 코드 영역을 건드리는 것 (테스트부터 추가).

### TDD 보강 규칙

- **TDD for Updates**: TDD 는 신규 코드만이 아니라 **모든 변경**에 적용된다. 기존 코드를 수정할 때도 새 동작을 표현하는 테스트를 먼저 추가/갱신해 실패시킨 뒤 구현한다.
- **TDD Coverage**: 테스트는 단순 assertion 이 아니라 **edge case 매트릭스**다. RED 단계에서 빈 입력·경계값·권한 실패·동시성 등을 빠짐없이 표현한다.
- **TDD Scope**: 컴포넌트/엔드포인트 단위로 사이클을 돈다. 프로젝트 전체 테스트를 한꺼번에 작성하지 않는다.
- **Bug = missing test**: 모든 버그는 그 케이스를 잡지 못한 테스트의 부재다. 버그 수정은 (1) 그 버그를 재현하는 실패 테스트, (2) 비슷한 결의 누락 케이스 추가, (3) 수정 순서로 진행한다.

## Edge Case 체크리스트 (RED 단계에서 반드시 점검)

기능을 쓸 때마다 아래 항목을 **의식적으로** 검토하고, 해당되는 케이스는 실패 테스트로 남긴다.

- [ ] **입력 검증**: null, 빈 문자열, 공백, 최대/최소 길이, 범위 밖 숫자, 허용되지 않는 문자
- [ ] **인증/권한**: 토큰 없음, 만료, 변조, 본인이 아님
- [ ] **리소스 상태**: 존재하지 않음, 이미 삭제됨, 중복 생성
- [ ] **경계값**: 0, 1, 최대-1, 최대, 최대+1
- [ ] **형식**: 잘못된 JSON/타입, 누락된 필드, 추가된 필드
- [ ] **부수효과**: 트랜잭션 롤백, 외부 상태 오염 없음
- [ ] **동시성** (해당 시): 동일 리소스 경쟁, 낙관적 잠금

## ZERO TOLERANCE — 테스트 실패의 소유권

테스트가 깨지는 순간, **그 실패는 작업자의 것**이다.

**금지 표현**:
- ❌ "이 실패는 내 변경과 무관하다"
- ❌ "이건 원래 깨져 있었다"
- ❌ "플레이키 테스트라 무시해도 된다"
- ❌ "이번엔 스킵하고 가자"

**필수 행동**:
- ✅ 실패 원인을 끝까지 추적한다
- ✅ 근본 원인을 수정한다 (테스트가 아니라 코드)
- ✅ 모든 테스트가 다시 초록불이 될 때까지 작업을 "완료" 로 표시하지 않는다

> **Rule**: 테스트가 깨졌을 때 "내 변경과 무관하다" 라는 것은 없다. 작업 세션 중 깨졌다면 그 사람이 고친다. 끝.

또한 테스트를 손쉽게 통과시키려고 **테스트 자체를 약화**시키면 안 된다:
- 어려운 코드 경로를 스킵하지 말고, 테스트하기 쉽게 코드를 분리한다.
- 인라인 콜백이 테스트 어렵다면 **이름 있는 함수로 추출**하고 그것을 직접 테스트한다.
- 커버리지 게이트를 낮추는 것은 마지막 수단이 아니라 **금지**다.

## 커밋 규칙

사이클의 각 단계는 **가능하면 별도 커밋**으로 분리한다. 최소한 RED 와 GREEN 은 분리한다.

```
test: <기능> — 실패 테스트 추가 (edge case 포함)
feat: <기능> 구현 — 테스트 통과
refactor: <영역> — <무엇을 개선했는지>
```

예시:
```
test: 회원가입 중복 username 거부 실패 테스트 추가
feat: AuthService.signup 중복 검증 구현
refactor: UserRepository.existsByUsername 활용으로 단순화
```

**커밋 컨벤션**:
- 제목 prefix: `feat`, `fix`, `test`, `refactor`, `docs`, `chore`, `ci`, `build`, `style`, `perf`
- 이슈 연결이 있으면 제목 또는 본문에 `#<번호>` 명시.
- 본문은 **왜** 를 설명. 무엇은 diff 가 보여 준다.
- **Amend 금지**: 새 커밋으로 누적한다. 강제 푸시·히스토리 재작성은 사용자 명시 승인 시에만.

## 테스트 구조 기준

- **통합 테스트 우선**: 시스템 계약(HTTP, CLI, 메시지)을 종단 간 검증하는 테스트를 기본으로.
- **모킹 최소화**: 외부 I/O(네트워크, 결제, 메일)만 모킹. DB 는 가능한 한 인메모리/도커 등 실제 동작.
- **테스트명**: 의도가 한 줄에 드러나야 한다. `given_when_then` 또는 `should_...` 또는 `<조건>_<결과>` 형식. 한국어 가능.
  - 예: `signup_중복된_username이면_400을_반환한다`
  - 예: `should_return_403_when_caller_is_not_author`
- **Arrange / Act / Assert** 구조를 지킨다.

## 작업 단위 제약 (이슈 · 파일 크기)

작은 단위로 쪼개야 리뷰 품질이 유지되고 TDD 사이클이 빠르게 돈다.

### 이슈당 수정 파일 수: **최대 10개**
- 하나의 이슈(PR)에서 건드리는 파일은 **10개 이하** 로 유지한다.
  - 프로덕션 코드, 테스트 코드, 설정 파일을 **모두 합쳐서** 카운트.
  - 빌드 도구가 자동 생성하는 파일(wrapper, lock 파일, 자동 생성 스크립트 등) 은 제외.
- 착수 전에 **예상 수정 파일 수를 세어 본다**. 10개를 초과할 것 같으면:
  1. **하위 이슈로 분할** 한다 (예: `2.0` → `2.1`, `2.2`).
  2. 분할 기준은 **독립적으로 TDD 한 사이클이 성립하는 feature slice**.
  3. 분할한 하위 이슈는 `x.1`, `x.2` 네이밍으로 새 이슈를 만들고 원래 이슈는 tracking(epic) 으로 유지하거나 닫는다.
- 10개를 넘기며 PR 을 올리지 않는다. 리뷰어가 한 번에 문맥을 잡을 수 있는 크기를 유지.

### 파일당 라인 수: **300줄 미만**
- 신규 파일은 **300줄 미만** 으로 작성한다.
- 기존 파일을 수정하다 300줄을 넘어갈 것 같으면:
  1. **책임 분리**를 먼저 고려 — 서비스/도메인/유틸을 별 클래스/모듈로 뽑는다.
  2. 테스트 파일이 커지면 **테스트 대상 단위로 클래스 분할**.
  3. 억지로 한 파일에 몰지 않는다.
- 자동 생성/보일러플레이트가 다수 포함되는 파일은 예외.

### 운영 체크리스트
- [ ] 착수 전: 이슈의 "범위/결과물" 에서 수정 파일 수 추정 → 10개 초과면 하위 이슈 제안
- [ ] 작업 중: 새 파일을 만들 때마다 한도를 상기
- [ ] PR 직전: `git diff --stat main...HEAD` 로 파일 수 확인, 300줄 초과 파일이 있으면 분할

## 커버리지

- **패치(변경분) 커버리지**: 가능한 한 **100%**, 게이트는 **최소 95%**. (프로젝트별로 `$COVERAGE_THRESHOLD` 변수로 정의)
- 커버리지를 핑계로 **테스트의 질을 낮추지 않는다**: 무의미한 한 줄 호출 테스트로 % 만 채우지 말고, edge case 를 표현하는 의미 있는 테스트로 채운다.
- 95% 게이트가 깨지면 **로컬에서 단일 최대 공백 클래스를 직접 exercise** 하는 단위 테스트가 보통 가성비가 가장 높다.

---

## 예외

- **실험용 스크래치 코드**: 저장소에 커밋하지 않으면 예외. 커밋하는 순간 TDD 적용.
- **설정/빌드 파일**: 테스트 대상 아님 (단, 로직이 들어 있다면 적용).
- **명백한 단순 DTO / Entity 필드만 있는 클래스**: 로직이 없으면 별도 테스트 불필요. 검증·계산·상태 전이가 들어가는 순간 TDD 적용.

## 한 줄 요약

> **테스트가 없으면 코드도 없다. RED 에서 실패를 설계하지 않으면 TDD 가 아니다.**

## 프로젝트 어댑터 (각 프로젝트에서 채울 변수)

heritage 의 스킬·문서는 도구 명령을 직접 박지 않고 아래 변수로 참조한다. 새 프로젝트에 들어갈 때 프로젝트 CLAUDE.md 또는 별도 `adapters.md` 에 다음 표를 채운다.

| 변수 | 의미 | 예시 |
|---|---|---|
| `$DEV_START_CMD` | 로컬 dev 서버 기동 명령 | `./gradlew bootRun` / `npm run dev` / `python manage.py runserver` |
| `$READY_PATTERN` | 기동 완료 로그 패턴 | `Started ApplicationName` / `ready in \d+ms` / `Listening on` |
| `$DEV_PORT` | 로컬 dev 서버 기본 포트 | `8080` / `3000` / `5173` |
| `$LINT_CMD` | 정적 검사 명령 | `./gradlew lint` / `npm run lint` / `ruff check` |
| `$TYPECHECK_CMD` | 타입 검사 명령 (해당 시) | `./gradlew compileJava` / `npm run typecheck` / `mypy .` |
| `$TEST_CMD` | 단위 테스트 명령 | `./gradlew test` / `npm test` / `pytest` |
| `$COVERAGE_CMD` | 커버리지 측정 명령 | `./gradlew test jacocoTestReport` / `npm run test:coverage` / `pytest --cov` |
| `$COVERAGE_THRESHOLD` | 커버리지 게이트 (게이트 도구 + 지표 + 값) | `JaCoCo INSTRUCTION 95%` / `Jest patch 95%` / `coverage.py 90%` |
| `$LOCAL_GATE_CMD` | "PR 올리기 전 한 방" 게이트 | `./gradlew clean build` / `npm run lint && npm test && npm run build` |
| `$RUNTIME_PATH` | 언어 런타임 PATH 익스포트 (필요 시) | `export PATH="/opt/homebrew/opt/openjdk@21/bin:$PATH"` |
| `$SOURCE_EXTS` | 변경 감지 정규식 | `\.java$` / `\.tsx?$\|\.css$` / `\.py$` |
| `$CI_TEST_JOB` | CI 테스트 잡 이름 | `Test + Coverage` / `unit-tests` |
| `$CI_BUILD_JOB` | CI 빌드 잡 이름 | `Build (no tests)` / `build` |
| `$E2E_CRED_ENV` (선택) | E2E 자격증명 환경변수 | `E2E_TEST_EMAIL`, `E2E_TEST_PASSWORD` (값은 절대 커밋 금지) |
| `$STORYBOOK_CMD` (선택) | 시각 컴포넌트 빌드/테스트 명령 | `npm run build-storybook` / `npm run test-storybook:ci` |
| `$INFRA_CHECK_CMD` (선택) | 로컬 인프라(DB/브로커) 상태 확인 | `supabase status` / `docker compose ps` |
| `$INFRA_START_CMD` (선택) | 로컬 인프라 기동 | `supabase start` / `docker compose up -d` |
| `$ENV_BOOTSTRAP_CMD` (선택) | 환경 파일 시드 명령 | `cp .env.example .env.local` |
| `$DEFAULT_BRANCH` | 메인 브랜치 이름 | `main` / `master` / `develop` |

스킬 본문에서 `$VAR` 가 보이면 위 표를 통해 프로젝트의 실제 명령으로 매핑된다. 어댑터 표가 비어 있다면 스킬은 "정의되지 않음" 으로 동작하고, 사용자에게 채워 달라고 요구한다.
