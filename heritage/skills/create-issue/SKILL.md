# create-issue — 버그/기능 묘사에서 이슈+드래프트 PR 까지

사용자의 자유 묘사를 받아, 코드 탐색·재현으로 사실을 굳힌 뒤 **이슈를 먼저 만들고**, **사용자 승인 게이트** 통과 후에야 TDD 구현·PR 로 넘어간다.

## 트리거
- 이슈 URL 없이 버그/기능을 묘사 ("로그인 후 대시보드가 깜빡인다", "프로필 페이지에 다크 모드가 필요해")
- "이슈 만들어줘", "이슈 파줘", "create issue", "file issue"
- 시각 증거(스크린샷) + 묘사

## CRITICAL Rule

**먼저 GitHub 이슈를 만든다(Step 0~3). 그 뒤 STOP 하고 사용자 승인을 기다린다(Step 4). 이슈 생성 없이 또는 사용자 승인 없이 구현으로 넘어가지 않는다. 예외 없음.**

## 워크플로우

### Step 1: 코드베이스 탐색

1. `CLAUDE.md` 적재 — 프로젝트 컨벤션·작업 단위 제약 인지.
2. 관련 파일 식별:
   - `Grep`/`Glob` 으로 키워드·심볼 위치
   - 광범위하면 `Explore` 서브에이전트 (병렬 1~3개)
3. 기존 구현·유틸 우선 — 중복 만들기 금지.

### Step 2: 버그 재현 (버그 한정 — MANDATORY)

**가정으로 이슈를 만들지 마라. 항상 재현 먼저.**

1. **dev 서버 기동** — `$DEV_START_CMD` 또는 `manual-test` 스킬 위임
2. **재현**:
   - 사용자 묘사대로 단계 진행
   - 실제 동작 관찰: snapshot, 네트워크, 콘솔, 서버 로그
3. **서버 로그** 확인: 에러, 실패 요청, 예상 외 동작
4. **브라우저 콘솔** 확인: 클라이언트 에러
5. **네트워크 탭** 확인: 실패 API 호출, 비정상 응답

**사실로 기록**:
- 실제 vs 기대
- 정확한 에러 메시지 (양쪽)
- 네트워크 요청/응답 세부
- 일관 vs 간헐

**중요**: 초기 가설(코드 읽기 기반)이 재현 결과와 다르면 분석을 갱신. 재현이 우선.

### Step 3: 분석 + 문서화

탐색 + 재현 종합:
- 영향 파일 식별
- (버그) 근본 원인 — 재현 증거 기반
- 기술 접근법
- **확정된 발견 vs 가설** 명시 구분

### Step 4: GitHub 이슈 생성 (FIRST — 승인 필요)

**중요**: 구현 전에 이슈를 만든다. 추적·PR 연결을 위해 필수.

```bash
gh issue create \
  --title "<type>(<scope>): <요약>" \
  --body-file /tmp/issue-body.md
```

**이슈 본문 형식** (한국어 템플릿):
```markdown
## 현상
- 현재 동작 vs 기대 동작

## 재현 결과 (Step 2 에서 관찰)
- 단계별 동작
- 에러 메시지 (서버·클라이언트)
- 네트워크/콘솔 발췌

## 영향 파일
- `경로/파일1`
- `경로/파일2`

## 수용 기준
- [ ] ...
- [ ] ...

## 기술 노트 (가설은 *가설* 로 표시)
- 확정: ...
- 가설: ...
```

**출력에서 이슈 번호를 캡처** (예: `#238`) — 드래프트 PR 에 사용.

### Step 5: 사용자 승인 게이트 (MANDATORY)

**여기서 STOP. 사용자 승인을 기다린다.**

이슈 생성 후:
1. 이슈 URL 을 사용자에게 제시
2. 제안 구현 접근법 요약
3. 수정 예상 파일 목록 (**≤10** 검증)
4. **명시적 승인을 요청**

`AskUserQuestion` 으로 확정:
- 질문: "이 이슈로 구현 진행할까요?"
- 옵션: "진행" / "이슈 먼저 검토" / "스코프 조정 필요"

**사용자가 명시적으로 진행을 승인할 때까지 Step 6 으로 가지 않는다.**

### Step 6: 구현 플랜 + TDD

`plan-issue` 의 Step 5~6 흐름을 그대로 사용:
- TDD 사이클 (RED → GREEN → REFACTOR)
- Edge case 체크리스트 적용
- 작업 단위 제약 (≤10 파일, <300줄/파일)

**버그 수정 시**:
- 버그를 잡는 실패 테스트 먼저
- 비슷한 결의 누락 케이스 식별
- 수정 (테스트 통과)

### Step 7: 로컬 검증

`local-tests` 스킬 위임. 모두 초록불이어야 다음 단계 진행.

### Step 8: UI 수동 검증 (UI 변경 시)

`manual-test` 스킬로 시연·증적.

### Step 9: 드래프트 PR

```bash
gh pr create --draft \
  --title "<type>(<scope>): #<issue-num> <요약>" \
  --body-file /tmp/pr-body.md
```

PR 본문은 `raise-pr` 표준 템플릿. 본문에 `Closes #<issue-num>` 명시.

## 출력

- GitHub 이슈 URL
- 통과 테스트가 동반된 구현
- 리뷰 대기 드래프트 PR URL

## 다른 스킬과의 관계

- **호출**: `manual-test` (Step 2 재현, Step 8 검증), `local-tests` (Step 7), `plan-issue` (Step 6 의 TDD 플랜 형식 차용 — 또는 직접 위임)
- **후속**: `raise-pr` (드래프트를 ready 로 전환·CI 감시)
- **범위 밖**: 이슈 URL 이 이미 주어진 경우 (`plan-issue` 사용), 다관점 리뷰 (`multi-agent-review`)
