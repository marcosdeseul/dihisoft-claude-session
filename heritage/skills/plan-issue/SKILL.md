# plan-issue — 이슈에서 TDD 플랜까지

GitHub 이슈를 받아 **재현 → 명확화 → TDD 플랜 → 구현 → 로컬 검증 → 드래프트 PR** 까지의 라이프사이클 진입점.

## 트리거
- GitHub 이슈 URL 또는 번호가 주어졌을 때 (예: `https://github.com/owner/repo/issues/42`, `#42`)
- "이 이슈 어떻게 풀지", "플랜 짜줘", "이슈 분석", "review issue", "plan issue"

## 워크플로우

### Step 1: 이슈 본문 수집

```bash
gh issue view <issue-url-or-number> --json title,body,labels,assignees,comments
```

추출:
- 이슈 제목 + 설명
- 라벨 (priority, type, component)
- 기존 코멘트의 추가 컨텍스트

### Step 2: 코드베이스 탐색

1. **`CLAUDE.md` 우선 적재** — 프로젝트 컨벤션·작업 단위 제약을 머리에 둔다.
2. 관련 파일 식별:
   - `Grep` 또는 `Glob` 으로 키워드/심볼 위치 파악
   - 범위가 넓으면 `Explore` 서브에이전트 호출 (병렬, 1~3개)
   - 주요 함수/클래스는 `Read` 로 직접 확인
3. 기존 구현·유틸 우선 — **새로 만들기 전에 재사용 가능한 것이 있는지** 확인.

### Step 3: 버그 재현 (버그 이슈 한정 — MANDATORY)

**가정으로 플랜을 짜지 마라.** 이슈 설명이 부정확하거나 불완전할 수 있다. 버그/UI 이슈라면 반드시 로컬 재현 먼저:

1. **dev 서버 기동** — 어댑터 `$DEV_START_CMD` 사용 또는 `manual-test` 스킬 위임
2. **재현 시도**:
   - 이슈에 적힌 단계 그대로 진행
   - 실제 동작 관찰: snapshot, 네트워크 요청, 콘솔 에러, 서버 로그
3. **사실 기반 기록**:
   - 실제 동작 vs 기대 동작
   - 정확한 에러 메시지 (서버·클라이언트 양쪽)
   - 네트워크 요청/응답
   - 일관 재현 vs 간헐 재현

**중요**: 재현 결과가 이슈 설명·코드 읽기 가설과 충돌하면 **관찰을 신뢰**한다. 분석을 갱신하라.

### Step 4: 명확화 질문

플랜 수립 전 모호함을 좁힌다 (`AskUserQuestion`):
- 요구사항이 불분명한 항목
- 사용자 결정이 필요한 기술 선택
- 스코프 경계 (이번 이슈에 포함/제외)
- 다른 이슈·외부 요인 의존
- **이슈 설명 vs 재현 결과의 불일치**

### Step 5: TDD 플랜 작성

CLAUDE.md 의 사이클·체크리스트와 정합되게:

1. **작업 분해**: 작은·테스트 가능한 단위로 분할.
2. **각 단위마다 TDD 설계**:
   - **기대 동작 분석**: 성공 케이스에서 무엇이 일어나야 하는가?
   - **Edge case 발굴** (CLAUDE.md 체크리스트 적용):
     - 입력 검증, 인증/권한, 리소스 상태, 경계값, 형식, 부수효과, 동시성
   - **RED**: 기대 동작 + edge case 를 표현하는 실패 테스트
   - **GREEN**: 최소 구현
   - **REFACTOR**: 단순화·중복 제거·네이밍·매직 넘버
3. **버그 수정**:
   - 재현 결과(Step 3)에 기반한 플랜
   - 버그를 잡는 실패 테스트 먼저
   - 비슷한 결의 누락 케이스 식별
4. **변경 파일**: 생성·수정 파일 목록 작성. **≤10개** 검증.
5. **네이밍 컨벤션**: 프로젝트 컨벤션 준수.
6. **데이터 변경**: 스키마/마이그레이션이 필요하면 명시.

### Step 6: TDD 구현

CLAUDE.md 사이클대로:
1. RED — 실패 확인 후
2. GREEN — 최소 구현
3. REFACTOR — 테스트 초록불 유지하며 정리

각 단계는 **별도 커밋** 권장.

### Step 7: 로컬 검증

`local-tests` 스킬 위임 — lint, typecheck, 단위 테스트, 커버리지, (해당 시) 시각/E2E.

### Step 8: UI 변경 시 수동 검증

UI 변경이 포함되면 `manual-test` 스킬로 시연·증적.

### Step 9: 드래프트 PR

```bash
gh pr create --draft \
  --title "<type>(<scope>): #<num> <요약>" \
  --body-file /tmp/pr-body.md
```

PR 본문은 `raise-pr` 의 표준 템플릿을 사용한다 (`Closes #N` + Summary + TDD 증적 + Test plan + 파일 표).

## 출력 형식

플랜은 다음을 포함:
- 이슈 요약 + 받은 명확화
- 순서 있는 작업 목록 (`TaskCreate` 통합)
- 영향받는 파일 목록 (≤10 검증)
- PR 전략 (≤10 파일, <300줄/파일)

## 가이드라인

- **재사용 우선**: 새 코드 작성 전 기존 구현 확인.
- **재현이 가설을 이긴다**: 코드 읽기 추론과 실제 재현이 다르면 재현이 정답.
- **이슈 본문 갱신**: 분석·플랜 결과를 이슈 코멘트로 남겨 다른 사람이 따라올 수 있게.
- **사용자 명확화 게이트**: Step 4 의 질문에 답을 받기 전에는 Step 5 로 진행하지 않는다.

## 다른 스킬과의 관계

- **호출**: `manual-test` (Step 3 재현 + Step 8 UI 검증), `local-tests` (Step 7), `raise-pr` (Step 9 PR 라이프사이클)
- **선행**: 없음 (라이프사이클 진입점)
- **후속**: `raise-pr` → `multi-agent-review`
- **범위 밖**: 머지, 다관점 리뷰, CI 디버깅
