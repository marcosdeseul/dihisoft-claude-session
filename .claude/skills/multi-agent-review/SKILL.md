# multi-agent-review — 세 관점 병렬 리뷰 + 교차 검증

하나의 PR을 **보안 / TDD-품질 / 아키텍처** 세 리뷰어가 각각 독립적으로 읽고, 메인 프로세서가 세 리포트를 병합해 **진짜 블로커**와 **관점별 고유 결함**을 분리해 제시한다.

## 핵심 가치

- **단일 관점 리뷰가 놓치는 교집합 리스크를 드러낸다** — 세 관점이 동시에 같은 결함을 지목하면 그것은 토론의 여지 없는 머지 블로커다.
- **관점별 격리** — 리뷰어끼리 스코프가 겹치지 않아 중복 노이즈가 없고, 한 에이전트가 모든 관점을 섞어 보는 것보다 각 관점의 깊이가 더 깊다.
- **컨텍스트 절약** — 메인 프로세서는 세 리포트의 원문을 요약 형태로만 소비하므로, 후속 작업을 위한 컨텍스트 윈도우가 비어 있다.

## 언제 사용하는가

- PR 머지 전 한 번의 컨텍스트에서 보안·품질·설계를 모두 검토
- 리뷰 결과로 후속 이슈를 쪼갤 근거가 필요할 때
- 자동화 테스트가 통과했지만 "이 PR이 진짜 준비됐는가"를 독립 관점으로 확인하고 싶을 때

**쓰지 말 것**:
- 한 관점만 필요 → `pr-reviewer` 에이전트 직접 호출
- 1~2파일짜리 자명한 수정 → 오버킬
- 아직 코드가 다 안 된 상태 → TDD 사이클을 마친 뒤 호출

## 전제

- 저장소 루트에 `CLAUDE.md` 존재 (각 리뷰어가 기준으로 삼음).
- `.claude/agents/pr-reviewer.md` 정의 존재 (이 스킬의 핵심 의존).
- `gh` CLI 인증 완료.
- PR 번호 **또는** 로컬 브랜치가 사용자 프롬프트 또는 현재 상태에서 식별 가능.

## 워크플로우

### 1. 대상 확정

사용자 프롬프트에서 PR 번호(`#21`, `PR 21`) 또는 브랜치명(`issue/8-jwt-login`)을 파싱한다. 둘 다 없으면:

```bash
gh pr view --json number,headRefName,title
```

현재 브랜치의 PR을 조회. PR이 없고 브랜치만 있다면 브랜치 diff(`git diff main...<branch>`)만으로도 진행. PR 번호를 못 찾으면 **사용자에게 확인** 후 진행.

### 2. 세 에이전트 병렬 실행 — **한 메시지에서 세 Agent 호출**

병렬이 이 스킬의 핵심이다. 반드시 **같은 응답 안에 세 개의 Agent tool call**을 넣어라. 순차 호출 금지.

각 호출:

```
Agent(
  subagent_type: "pr-reviewer",
  description: "<scope> review of PR #<N>",
  prompt: """
Review PR #<N> on branch `<branch>` in `<absolute repo path>`.
scope=<security|tdd-quality|architecture>
Context: <1~2문장 — 기능 이름, 이 PR이 다루는 slice, 참조할 이전 PR/이슈>.

Follow the rules in your agent definition. Return the full markdown report.
"""
)
```

세 호출의 `scope`만 다르고 나머지 context·path·PR 번호는 동일. 세 리뷰어가 **서로의 결과를 모른 채 독립적으로** 판단하도록 한다 — 메인 프로세서의 의견이나 한쪽 리뷰어의 결론을 다른 리뷰어 프롬프트에 섞지 마라. 편향 유발.

### 3. 교차 검증 (cross-check)

세 리포트가 돌아오면 다음 표를 직접 작성:

| # | 이슈 요약 | 보안 | TDD | 아키텍처 |
|---|-----------|:---:|:---:|:---:|
| 1 | ... | #finding# | — | #finding# |

- **둘 이상에서 지목된 항목 = 진짜 리스크 (P0 후보)**.
- 한 관점 단독 = 그 관점의 고유 관찰 (P1/P2 후보).
- 표기는 각 리뷰어 리포트의 finding 번호를 그대로 인용.

### 4. 우선순위별 개선안으로 정리

사용자에게 반환할 최종 포맷:

```
# 종합 리뷰 — PR #<N>

## 세 관점이 동시에 지목한 문제 (= 진짜 리스크)
<교차표>

## 우선순위별 개선안
### P0 — 머지 블로커 (본 PR 또는 즉시 후속 hotfix)
1. ...

### P1 — 본 PR 범위로 같이 넣거나 바로 다음 PR
...

### P2 — 후속 이슈로 분리 (CLAUDE.md 10-파일 제약 고려)
...

## 세 리뷰어가 공통으로 칭찬한 점
...

## 한 줄 결론
```

**CLAUDE.md 10-파일 제약**을 의식적으로 적용 — 제안 묶음이 한도를 넘으면 하위 이슈 분할안까지 본문에 포함.

### 5. 종료

- 세 리뷰어의 원문 리포트는 **요청이 있을 때만** 전체 공개. 기본은 종합 리뷰만.
- 이슈 생성·코드 수정·PR 코멘트는 **별도 턴** 또는 다른 스킬(`raise-pr`, `plan-issue`).

## 가이드라인

- **세 Agent 호출은 반드시 한 메시지 = 진짜 병렬.** 순차 호출은 시간·토큰 낭비이고 어차피 독립 판단이 목적이므로 순서 의미 없음.
- **스코프 침범 감시**: 리뷰어 정의에 박혀 있지만, 병합 시 같은 결함이 두 리포트에 동시에 등장하면 **원주인 스코프로 귀속**하고 다른 쪽에서는 언급만 (교차표의 가치는 오히려 더 커진다).
- **중립적 context 전달**: 메인 프로세서의 선입견을 프롬프트에 섞지 마라 ("이 PR은 시크릿 관리가 허술해 보이는데 확인해줘" 금지 — 리뷰어가 그 지점만 보게 된다).
- **제안을 만들 때 CLAUDE.md 규칙 적용**: 파일 수, 커밋 분리, RED-first 원칙 등.
- **사용자가 "원문 보고서"를 요구하면** 세 리포트를 순서대로(보안 → TDD → 아키텍처) 그대로 덤프.

## 범위 밖

- 이슈 생성 — `gh issue create`는 별도 턴 또는 후속 스킬.
- 코드 수정 · 테스트 실행 · PR 코멘트 작성 — 이 스킬은 **리뷰 결과물만** 책임.
- 단일 관점 리뷰 — `pr-reviewer` 에이전트 직접 호출로 충분.
- 비-PR 전체 감사 — 다른 감사 스킬(`geo-audit` 등) 또는 general-purpose.

## 예시 실행 — PR #21

**사용자**: "PR #21 멀티 에이전트 리뷰 돌려줘."

**메인 프로세서**:
1. `gh pr view 21 --json headRefName,title` → branch=`issue/8-jwt-login`, title=`JWT 로그인`
2. 한 메시지에서 세 Agent tool call 동시 호출:
   - `subagent_type=pr-reviewer`, `scope=security`
   - `subagent_type=pr-reviewer`, `scope=tdd-quality`
   - `subagent_type=pr-reviewer`, `scope=architecture`
   각 프롬프트의 Context는 동일: "JWT 인프라 + 로그인 API + 로그인 UI를 도입하는 PR. 이전 PR #7(회원가입)에 이어짐."
3. 세 리포트 수신 → 공통/고유 finding 표로 교차 검증 → P0/P1/P2 개선안 작성.
4. 사용자에게 종합 리포트만 제시. 원문은 저장해 두고 요청 시 공개.
