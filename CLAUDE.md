# CLAUDE.md

이 저장소에서 Claude Code(및 모든 기여자)가 코드를 작성할 때 반드시 따라야 할 원칙을 정의한다.

---

## 개발 철학: TDD 필수

이 프로젝트의 **모든 프로덕션 코드는 TDD 기반으로 작성**한다.
TDD는 단순히 "테스트 코드를 먼저 쓰는 것"이 아니라,
**RED 단계에서 시스템에 발생할 수 있는 모든 edge case를 선제적으로 식별하고 커버하는 개발 철학**이다.

### 사이클: RED → GREEN → REFACTOR

#### 1. RED — "실패하는 테스트를 먼저 쓴다"
- **목적**: 이 단계는 테스트 작성이 목적이 아니다. **요구사항과 시스템 경계를 파악**하는 단계다.
- **해야 할 것**:
  - Happy path뿐 아니라 **edge case를 적극적으로 발굴**한다.
  - 예: 입력 검증 실패, 빈 값/null/경계값, 중복, 권한 없음, 만료된 토큰, 동시성, 존재하지 않는 리소스, 허용되지 않은 상태 전이, 잘못된 타입/포맷.
  - 각 edge case마다 **명시적으로 실패하는 테스트**를 작성한다.
  - 테스트는 반드시 **먼저 실패(RED)** 해야 한다. 통과한다면 테스트가 틀렸거나 구현이 이미 있다는 뜻.
- **하지 말 것**:
  - "나중에 edge case 추가하자"며 happy path만 쓰고 다음 단계로 넘어가지 않는다.
  - 프로덕션 코드를 먼저 생각하고 거기에 맞춰 테스트를 쓰지 않는다(역방향 TDD 금지).

#### 2. GREEN — "테스트를 통과시키는 최소 코드를 쓴다"
- **목적**: 오직 RED 테스트들을 통과시키는 것만 목표.
- **해야 할 것**:
  - 가능한 한 단순하게, 직관적인 구현.
  - 실패 테스트가 없는 기능은 이 단계에서 추가하지 않는다.
- **하지 말 것**:
  - 미래를 위한 추상화, 사용되지 않는 매개변수, 아직 테스트 없는 분기 추가 금지.
  - "어차피 필요할 것 같아서"로 코드를 넣지 않는다.

#### 3. REFACTOR — "동작을 바꾸지 않고 구조를 개선한다"
- **목적**: 테스트가 초록불인 상태에서 **내부 구조만** 다듬는다.
- **해야 할 것**:
  - 중복 제거, 명명 개선, 함수/클래스 분리, 매직 넘버 상수화.
  - 리팩터 중 **모든 테스트는 계속 통과**해야 한다. 한 번이라도 깨지면 즉시 되돌린다.
- **하지 말 것**:
  - 기능 추가. 새 동작이 필요하면 사이클을 처음(RED)부터 다시 돈다.
  - 테스트가 없는 코드 영역을 건드리는 것(테스트부터 추가).

## 커밋 규칙

사이클의 각 단계는 **가능하면 별도 커밋**으로 분리한다. 최소한 RED와 GREEN은 분리한다.

```
test: <기능> — 실패 테스트 추가 (edge case 포함)
feat: <기능> 구현 — 테스트 통과
refactor: <영역> — <무엇을 개선했는지>
```

예시:
```
test: 회원가입 중복 username 거부 실패 테스트 추가
feat: AuthService.signup 중복 검증 구현
refactor: UserRepository.existsByUsername 활용으로 단순화
```

## Edge Case 체크리스트 (RED 단계에서 반드시 점검)

기능을 쓸 때마다 아래 항목을 **의식적으로** 검토하고, 해당되는 케이스는 실패 테스트로 남긴다.

- [ ] **입력 검증**: null, 빈 문자열, 공백, 최대/최소 길이, 범위 밖 숫자, 허용되지 않는 문자
- [ ] **인증/권한**: 토큰 없음, 만료, 변조, 본인이 아님
- [ ] **리소스 상태**: 존재하지 않음, 이미 삭제됨, 중복 생성
- [ ] **경계값**: 0, 1, 최대-1, 최대, 최대+1
- [ ] **형식**: 잘못된 JSON, 잘못된 타입, 누락된 필드, 추가된 필드
- [ ] **부수효과**: 트랜잭션 롤백, 외부 상태 오염 없음
- [ ] **동시성** (해당 시): 동일 리소스 경쟁, 낙관적 잠금

## 테스트 구조 기준

- **통합 테스트 우선**: Controller 레이어는 `@SpringBootTest` + `MockMvc`로 HTTP부터 DB까지 실제 경로 검증.
- **모킹 최소화**: 외부 I/O(네트워크 등)만 모킹. Repository는 H2 인메모리로 실제 동작 검증.
- **테스트명 규칙**: 한국어로 의도 명시 — `given-when-then` 또는 `should_...` 형식.
  - 예: `signup_중복된_username이면_400을_반환한다`
- **Arrange / Act / Assert** 구조를 지킨다.

## 작업 단위 제약 (이슈 · 파일 크기)

작은 단위로 쪼개야 리뷰 품질이 유지되고 TDD 사이클이 빠르게 돈다.

### 이슈당 수정 파일 수: **최대 10개**
- 하나의 이슈(PR)에서 건드리는 파일은 **10개 이하**로 유지한다.
  - 프로덕션 코드, 테스트 코드, 설정 파일을 **모두 합쳐서** 카운트.
  - Gradle wrapper 자동 생성 파일은 제외.
- 착수 전에 **예상 수정 파일 수를 세어본다**. 10개를 초과할 것 같으면:
  1. **하위 이슈로 분할**한다 (예: `2.0` → `2.1`, `2.2`).
  2. 분할 기준은 **독립적으로 TDD 한 사이클이 성립하는 feature slice**. 엔티티 하나, API 하나 단위가 좋다.
  3. 분할한 하위 이슈는 `x.1`, `x.2` 네이밍으로 새 이슈를 생성하고, 원래 이슈는 tracking(epic)으로 유지하거나 닫는다.
- 10개를 넘기며 PR을 올리지 않는다. 리뷰어가 문맥을 한 번에 파악할 수 있는 크기를 유지.

### 파일당 라인 수: **300줄 미만**
- 신규 파일은 **300줄 미만**으로 작성한다.
- 기존 파일을 수정하다 300줄을 넘어갈 것 같으면:
  1. **책임 분리**를 먼저 고려 — 서비스/도메인/유틸을 별 클래스로 뽑는다.
  2. 테스트 파일이 커지면 **테스트 대상 단위로 클래스 분할** (예: `AuthControllerTest` → `AuthSignupControllerTest`, `AuthLoginControllerTest`).
  3. 억지로 한 파일에 몰지 않는다.
- 자동 생성/보일러플레이트가 포함되는 파일(빌드 스크립트, wrapper 등)은 예외.

### 운영 체크리스트
- [ ] 착수 전: 이슈의 "범위/결과물"에서 수정 파일 수 추정 → 10개 초과면 하위 이슈 제안
- [ ] 작업 중: 새 파일을 만들 때마다 한도를 상기
- [ ] PR 직전: `git diff --stat main...HEAD` 로 파일 수 확인, 300줄 초과 파일 있으면 분할

## 예외

- **실험용 스크래치 코드**: 리포지토리에 커밋하지 않으면 예외. 커밋하는 순간 TDD 적용.
- **설정/빌드 파일**(`build.gradle.kts`, `application.yml` 등): 테스트 대상 아님.
- **명백한 단순 DTO / Entity 필드만 있는 클래스**: 로직이 없으면 별도 테스트 불필요. 로직(검증, 계산, 상태 전이)이 들어가는 순간 TDD 적용.
